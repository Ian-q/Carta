Routing Approaches: How much classification is enough?

Two options — differ mainly in how we handle flattened pages and figure detection

A

Metadata-only routing (recommended)

PyMuPDF tells us everything we need — no model for routing, only for extraction

text = page.get_text() → long?
images = page.get_images() → any?
tables = column-alignment heuristic
captions = regex("fig|figure|plot|chart")

if text_long and no_images and no_tables:
  → PyMuPDF only (skip vision)
elif text_long and has_tables:
  → GLM-OCR (structured)
elif text_long and (has_images or has_captions):
  → PyMuPDF text + LLaVA on image crops
else (no/little text):
  → GLM-OCR (handles OCR + structure)
    if yield low → LLaVA describe
Zero model calls for routing Figure captions = free heuristic Flattened: GLM-OCR runs blind
B

Metadata + small classifier for flattened pages

Same as A, but flattened pages get a cheap multimodal "what is this?" scan before deciding

# same as A for text-bearing pages...

else (no/little text):
  # Ask small model: text/table/diagram/photo?
  kind = moondream.classify(page_image)
  if kind == "text" or "table":
    → GLM-OCR (structured OCR)
  elif kind == "diagram":
    → LLaVA (description)
  else (photo/decorative):
    → skip (low value)
Avoids OCR on decorative photos Better routing for scanned PDFs New model dependency (moondream/minicpm) Extra latency on every flattened page

Figure caption detection (in both approaches) — scan page.get_text() for patterns like "Fig. 3", "Figure 12: Voltage vs time", "see plot above". If found and no image in PyMuPDF metadata, it suggests either: (a) the image is a vector graphic not listed by get_images(), or (b) the figure is on a nearby page. Could trigger a LLaVA pass on the full page render as a safety net.

My recommendation is A — GLM-OCR handles flattened pages well enough, and approach B adds a new model dependency for a relatively rare case. We can add B later if flattened-page quality is a real issue. Does A feel right?
