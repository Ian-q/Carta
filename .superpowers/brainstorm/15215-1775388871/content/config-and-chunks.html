Design Section 3: Config & Chunk Output

New tunable thresholds and what each page class produces in Qdrant

New config keys (under embed: in .carta/config.yaml)

embed:
  # existing — unchanged
  ocr_model: glm-ocr:latest
  ollama_vision_model: llava:latest
  ollama_url: http://localhost:11434

  # new thresholds
  vision_text_min_chars: 150
    # below → FLATTENED
  vision_text_max_chars: 600
    # above → captions are cross-refs
  vision_flattened_min_yield: 50
    # GLM-OCR chars below → try LLaVA
  vision_max_images_per_page: 4
    # cap LLaVA calls (skips tiny icons)
All have sensible defaults — existing projects don't need to change their config. vision_routing: auto (existing key) is removed/ignored.

Chunk output per page class

PURE_TEXT → 0 image chunks
PyMuPDF text chunk already created by pipeline. Vision returns empty list.
STRUCTURED_TEXT → 1 chunk
doc_type: "image_description"
model_used: "glm-ocr"
page_class: "structured_text"
image_index: 0
TEXT_WITH_IMAGES → 1 chunk per image crop
doc_type: "image_description"
model_used: "llava"
page_class: "text_with_images"
image_index: 0, 1, 2… (per crop)
# PyMuPDF text chunk kept separately
FLATTENED → 1 chunk
doc_type: "image_description"
model_used: "glm-ocr" or "llava"
page_class: "flattened"
image_index: 0
Backward compatibility: doc_type: "image_description" is preserved on all chunks so existing Qdrant collections don't need to be re-indexed. The new page_class field is additive — existing search queries ignore it, but it's available for future filtering (e.g. "only return chunks from structured table pages").

vision_max_images_per_page = 4 prevents pages with many small decorative icons (arrows, warning symbols, pin diagrams with 20 tiny images) from triggering 20 LLaVA calls. Images beyond the cap are skipped. Largest images first via bounding box area sort.
