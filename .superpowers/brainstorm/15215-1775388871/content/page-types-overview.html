PDF Page Classification: The 5 Cases

Mapping out what you described — confirm this is the right taxonomy before we design the system

📄

Case 1: Pure Text

PyMuPDF extracts text cleanly. No images present.

Action: PyMuPDF only — skip all vision models
✓ Fast, cheap, already works well
📊

Case 2: Structured Text / Tables

Text is present but arranged in tables — row/column relationships matter.

Action: GLM-OCR (structured extraction) to preserve table alignment
⚠ PyMuPDF may scramble row associations
🖼️

Case 3: Flattened Image Page

Entire page is a raster image. Could be: scanned text, scanned table, diagram, or decorative photo.

Sub-routing: GLM-OCR first → if low yield, LLaVA for description
⚠ Can't distinguish text-scan vs photo without running a model
📝🔬

Case 4: Text + Technical Image

Good PyMuPDF text AND an embedded image that looks like a plot, timing diagram, or schematic.

Action: Keep PyMuPDF text chunk + extract image crop + LLaVA describe image
✓ This is the key case carta was missing
📝🌳

Case 5: Text + Non-Technical Image

Good PyMuPDF text AND an embedded image that's decorative/irrelevant (logo, stock photo, chapter header art).

Action: Keep PyMuPDF text only — skip image description (it adds noise, not signal)
✗ Key question: how do we tell Case 4 from Case 5 cheaply?
