# Phase 999.3: Qdrant Collection Scoping — Multi-Platform Context Sharing

**Phase ID:** 999.3-qdrant-collection-scoping-multi-platform
**Status:** Planning
**Depends on:** 999.1 (sidecar taxonomy), 999.2 (vision pipeline)
**Created:** 2026-04-01

---

## Problem Statement

Currently, Carta searches are limited to the current repository's collections only. While this prevents cross-contamination between projects, it also means that valuable cross-project knowledge (e.g., a quirk solved in a UI project that applies to a new backend project) cannot surface automatically.

Additionally, Carta was originally designed for Claude Code only. With the rise of other agentic coding platforms like **OpenCode**, Carta needs to support multiple AI assistant platforms equally.

---

## Goals

1. **Repository Isolation (Default):** By default, queries are scoped to the current repository's collections only — no cross-contamination between projects
2. **Opt-in Cross-Project Recall:** Users can enable `cross_project_recall` to search across multiple project collections
3. **Global Memory Pool:** Support a `carta_global_*` collection namespace for explicitly promoted, project-agnostic memories
4. **Multi-Platform Support:** Carta works seamlessly with both **Claude Code** and **OpenCode** (and other MCP-compatible platforms)

## Collection Naming Strategy

### Current Behavior (Preserved)
```
{project_name}_doc       # General documents
{project_name}_notes     # Quirks, bug-notes, helpful-notes
{project_name}_session   # Session memory
```

### New: Global Pool
```
carta_global_doc         # Project-agnostic documents
carta_global_notes       # Global quirks and patterns
carta_global_session     # Shared session context
```

### New: Scoped Search Collections

**Scope: `repo` (default)**
- Query: `{project_name}_*`
- Collections: `{project_name}_doc`, `{project_name}_notes`, `{project_name}_session`

**Scope: `shared`**
- Query: All accessible project collections + global
- Filtered by `cross_project_recall.project_filter`

**Scope: `global`**
- Query: `carta_global_*` only

## Configuration Extensions

The existing `cross_project_recall` block will be extended:

```yaml
# Current (999.1)
cross_project_recall:
  enabled: false
  scope: ["quirk"]
  require_ollama_judge: true
  project_filter:
    mode: "all"  # or "include", "exclude"
    projects: []

# New (999.3)
cross_project_recall:
  enabled: false
  scope: ["quirk"]
  require_ollama_judge: true
  project_filter:
    mode: "all"  # or "include", "exclude"
    projects: []
  default_search_scope: "repo"  # "repo" | "shared" | "global"
  global_pool:
    enabled: true
    auto_promote: false  # Auto-promote high-value chunks
    promotion_threshold: 0.95  # Minimum score for auto-promotion
```

## Key Design Decisions

### 1. Collection Discovery

Qdrant doesn't have a native "list collections by prefix" API. We'll use:
- **Option A:** Maintain a registry in `~/.carta/collection-registry.yaml`
- **Option B:** Query Qdrant `/collections` endpoint and filter by prefix
- **Selected:** Option B (simpler, no state management)

### 2. Cross-Project Security

- By default, projects cannot see each other's collections
- `cross_project_recall.enabled: true` explicitly opts in
- `project_filter` provides whitelist/blacklist control
- Global collections are readable by all (by design)

### 3. Promotion to Global Pool

Two mechanisms:
1. **Manual:** MCP tool `carta_promote(chunk_id, target="global")`
2. **Automatic:** High-relevance chunks (score > threshold) auto-promoted

## Out of Scope (for 999.3)

| Feature | Reason |
|---------|--------|
| Cross-repo write operations | Security complexity; read-only cross-project for now |
| Real-time sync between projects | Eventually consistent is acceptable |
| Collection-level ACLs | File-system permissions suffice for local tool |
| Cloud-hosted shared collections | Local-only design principle |

---

## Research Notes

- Qdrant collection listing: `GET /collections` returns all collection names
- MCP spec is platform-agnostic; both Claude Code and OpenCode implement it
- OpenCode uses `.opencode.json` for MCP server registration (similar to `.mcp.json`)
- Collection scoping is a Carta-side concern, not Qdrant-side (no native multi-tenant support needed)
