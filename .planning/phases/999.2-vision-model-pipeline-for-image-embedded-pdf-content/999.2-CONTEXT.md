# Phase 999.2: Vision Model Pipeline for Image-Embedded PDF Content ‚Äî Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Extract visual content from PDFs using a local vision model via Ollama ‚Äî describing charts, diagrams, register maps, and figures that contain no text layer. Image descriptions become first-class embedded chunks, searchable alongside text content.

Two deliverables:
1. Image extraction from PDFs (embedded image objects + page-render fallback for vector-drawn diagrams)
2. Vision model pipeline: image bytes ‚Üí Ollama ‚Üí text description ‚Üí chunk ‚Üí Qdrant

This phase does NOT change the hook, search, or scan subsystems ‚Äî it extends the embed pipeline only.

</domain>

<decisions>
## Implementation Decisions

### Vision Model
- **D-01:** Config-driven via new key `ollama_vision_model` (default: `llava:latest`)
- **D-02:** LLaVA is the default ‚Äî accuracy over speed, since image descriptions serve as source-of-truth references in search results
- **D-03:** Model key follows the existing `ollama_model` pattern in `carta/config.py:DEFAULTS`

### Image Detection & Extraction
- **D-04:** Always process any page containing at least one image ‚Äî no threshold tuning
- **D-05:** Two extraction modes (both active, not mutually exclusive per page):
  1. **Embedded images:** `doc.extract_image(xref)` at native resolution, cap any dimension at 2048px (aspect-ratio-preserving scale-down for outliers)
  2. **Page render fallback:** For pages with no embedded image objects but with vector drawing content detected (shapes/paths via PyMuPDF), render the page at 150 DPI
- **D-06:** Always convert to PNG for delivery to Ollama ‚Äî avoids re-compressing already-lossy JPEG content; ensures lossless delivery

### Vision Prompt
- **D-07:** Technical extraction prompt (not generic description):
  - Instructs model to extract: data values, axis labels, units, register field names and bit widths, table contents, pin names, waveform annotations
  - Appropriate for datasheets, timing diagrams, temperature curves, block diagrams
  - Prompt text defined as a constant in the new vision module (not configurable per D-01‚ÄìD-03 scope)

### Chunk Integration
- **D-08:** Separate chunks per image ‚Äî each image description becomes its own Qdrant chunk
- **D-09:** Chunk metadata: `doc_type: "image_description"`, `page_num: N`, `image_index: N` (index of image on the page)
- **D-10:** These chunks use the same `upsert_chunks()` pipeline as text chunks (same embed model: `nomic-embed-text`)

### Failure Handling
- **D-11:** Fail-open ‚Äî if vision model is unavailable or returns an error for a specific image, skip that image and log a warning; text embedding of the PDF continues unaffected
- **D-12:** Consistent with HOOK-05 fail-open philosophy established in Phase 5

### Sidecar Updates
- **D-13:** Add two new fields to the sidecar schema (extends Phase 999.1 schema):
  - `image_count: N` ‚Äî number of images detected across all pages
  - `image_chunks: N` ‚Äî number of vision description chunks successfully embedded
- **D-14:** If `image_count > 0` and `image_chunks == 0`, the sidecar `status` remains `embedded` but the discrepancy is a signal that the vision model was unavailable during embed

### Claude's Discretion
- Module location: new `carta/embed/vision.py` following `parse.py` pattern ‚Äî Claude decides internal function signatures
- Whether `extract_image` and page-render fallback live in `vision.py` or extend `parse.py` ‚Äî recommend new module for separation of concerns
- Exact technical extraction prompt wording ‚Äî keep focused on data extraction, ‚â§200 chars to stay within small-model context
- Whether to check `ollama_vision_model` availability at startup (like `_check_ollama`) or lazily at first image ‚Äî lazy is fine

</decisions>
