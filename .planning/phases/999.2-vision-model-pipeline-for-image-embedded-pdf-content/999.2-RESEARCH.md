## Summary

Phase 999.2 extends the carta embed pipeline to process visual content in PDFs. The scope is narrow and well-bounded: a new `carta/embed/vision.py` module handles image extraction and Ollama vision calls, while `_embed_one_file()` in `pipeline.py` calls it after text extraction and folds the resulting image description chunks into the same `upsert_chunks()` flow.

PyMuPDF 1.27.2 (installed on this machine) provides the full required API: `page.get_images()` for embedded image detection, `doc.extract_image(xref)` for native-resolution extraction, `page.get_drawings()` for vector-content detection, and `page.get_pixmap(dpi=150)` for page-render fallback. The Ollama `/api/generate` endpoint accepts images as base64-encoded strings in an `images` array — the same base URL as the existing text generation calls in `hook.py`.

The primary implementation risk is CMYK colorspace handling: `extract_image()` returns raw bytes in native colorspace, which cannot be saved as PNG if CMYK. The safe path is always to route through `fitz.Pixmap` for colorspace normalization before PNG conversion.

**Primary recommendation:** Implement `carta/embed/vision.py` as a self-contained module with `extract_image_descriptions(pdf_path, cfg) -> list[dict]` as the public entry point. Extend `_embed_one_file()` in `pipeline.py` to call it for `.pdf` files and merge image description chunks into the upsert batch.

### Pattern 1: Image Extraction from PDF Page (Embedded Images)

Use `page.get_images()` to enumerate xrefs, then `doc.extract_image(xref)` for raw bytes. Always go through `fitz.Pixmap` for CMYK-safe PNG output.

```python
# Source: PyMuPDF docs + verified PyMuPDF 1.27.2 API
import fitz
import io

def _extract_embedded_images(doc: fitz.Document, page: fitz.Page) -> list[bytes]:
    """Extract embedded image bytes from a page as PNG. Returns list of PNG bytes."""
    results = []
    for img_info in page.get_images(full=True):
        xref = img_info[0]
        try:
            pix = fitz.Pixmap(doc, xref)
            # Convert CMYK or other non-RGB colorspaces to RGB before PNG
            if pix.n > 4:  # CMYK has n=4, alpha-CMYK has n=5
                pix = fitz.Pixmap(fitz.csRGB, pix)
            elif pix.n == 4 and pix.alpha:
                pix = fitz.Pixmap(fitz.csRGB, pix)
            # Cap dimensions at 2048px (aspect-ratio preserving)
            if pix.width > 2048 or pix.height > 2048:
                scale = 2048 / max(pix.width, pix.height)
                mat = fitz.Matrix(scale, scale)
                pix = pix.transform(mat)  # NOTE: use page.get_pixmap with matrix instead for page-level
            png_bytes = pix.tobytes("png")
            results.append(png_bytes)
        except Exception as e:
            print(f"Warning: skipping xref {xref} — {e}", file=sys.stderr, flush=True)
    return results
```

**Scaling note:** `fitz.Pixmap` does not have a `transform()` method. For embedded images needing resize, use `fitz.Pixmap(doc, xref)` then check dimensions; if over 2048, re-render via `page.get_pixmap(matrix=fitz.Matrix(scale, scale))` clipped to that image's bbox, or use PIL/BytesIO — but since PIL is not a dependency, the simplest approach is to use `Pixmap` and scale by creating a new Pixmap at the target size: `pix = pix.scale(new_w, new_h)` (PyMuPDF 1.18+ has `Pixmap.scale()`). Confirm with: `python3 -c "import fitz; print([m for m in dir(fitz.Pixmap) if 'scale' in m.lower()])"`.

### Pattern 2: Page Render Fallback (Vector Drawings)

Use `page.get_drawings()` to detect vector content. If the page has drawings but no embedded image objects, render the whole page as a pixmap.

```python
# Source: PyMuPDF docs recipes-images + drawing detection
def _has_vector_content(page: fitz.Page) -> bool:
    """Return True if page contains vector drawing paths (shapes, lines, curves)."""
    return len(page.get_drawings()) > 0

def _render_page_as_png(page: fitz.Page, dpi: int = 150) -> bytes:
    """Render entire page to PNG bytes at specified DPI."""
    pix = page.get_pixmap(dpi=dpi)
    return pix.tobytes("png")
```

### Pattern 3: Ollama Vision API Call

Endpoint: `POST {ollama_url}/api/generate` with `images` array of base64 strings.

```python
# Source: Ollama API docs (github.com/ollama/ollama/blob/main/docs/api.md)
import base64
import requests

VISION_PROMPT = (
    "Extract all technical data from this image: data values, axis labels and units, "
    "register field names and bit widths, table cell contents, pin names, "
    "waveform annotations, and block labels. List each item."
)  # 196 chars — within ≤200 target

def _call_vision_model(
    image_png_bytes: bytes,
    ollama_url: str,
    model: str,
    timeout: int = 60,
) -> str:
    """Send PNG bytes to Ollama vision model. Returns description text or raises."""
    b64 = base64.b64encode(image_png_bytes).decode("utf-8")
    resp = requests.post(
        f"{ollama_url}/api/generate",
        json={
            "model": model,
            "prompt": VISION_PROMPT,
            "images": [b64],
            "stream": False,
        },
        timeout=timeout,
    )
    if resp.status_code != 200:
        raise RuntimeError(f"Ollama vision error ({resp.status_code}): {resp.text[:200]}")
    return resp.json()["response"].strip()
```

### Anti-Patterns to Avoid

- **Using `doc.extract_image(xref)["image"]` bytes directly for PNG**: `extract_image()` returns raw bytes in the image's native format (JPEG, CMYK, etc.). It cannot be PNG-saved if CMYK. Always route through `fitz.Pixmap(doc, xref)` for colorspace normalization.
- **Checking `img["ext"]` to skip non-image xrefs**: `page.get_images()` already filters to image xrefs — `img[0]` is always an image xref.
- **Setting `stream: True` on Ollama vision call**: Streaming requires response accumulation. Use `stream: False` for simplicity, matching the existing `hook.py` pattern.
- **Importing fitz at module level**: `parse.py` does `import fitz` inside the function. Follow this pattern in `vision.py` to keep import lazy and testable.
- **Running vision on every PDF page unconditionally**: Only run on pages where `page.get_images()` returns results OR `page.get_drawings()` is non-empty.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CMYK→RGB colorspace conversion | Custom byte-level conversion | `fitz.Pixmap(fitz.csRGB, cmyk_pix)` | PyMuPDF handles ICC profiles correctly |
| Image resize/downscale | PIL or manual byte manipulation | `fitz.Matrix(scale, scale)` in `get_pixmap()` or `Pixmap` constructor | Built into PyMuPDF, no new deps |
| PNG encoding of extracted image | `struct.pack` or pillow | `pix.tobytes("png")` | Single-line, correct, no deps |
| Base64 encoding | Custom hex/byte loop | `base64.b64encode(bytes).decode()` | stdlib |
| Vision model HTTP call | Custom async client | `requests.post(..., timeout=60)` | Matches existing embed.py / hook.py pattern |
| Vector-drawing detection | Custom path parser | `page.get_drawings()` | PyMuPDF built-in, returns all path objects |

**Key insight:** PyMuPDF handles all image format complexity. The vision pipeline needs zero image processing libraries beyond what is already installed.

## Common Pitfalls

### Pitfall 1: CMYK PNG Conversion Failure

**What goes wrong:** `fitz.Pixmap(doc, xref).tobytes("png")` raises or produces corrupted PNG if the embedded image is CMYK (common in professional/print PDFs and datasheets).

**Why it happens:** PNG format requires RGB or grayscale. `pix.n` is 4 for CMYK (no alpha) or 5 for CMYK+alpha. The check `pix.n < 5` is insufficient — CMYK has `n=4` and no alpha, which passes the naive `< 5` check but still fails PNG output.

**How to avoid:** Check `pix.colorspace` not just `pix.n`. Convert if not already `fitz.csGRAY` or `fitz.csRGB`:
```python
if pix.colorspace not in (fitz.csGRAY, fitz.csRGB):
    pix = fitz.Pixmap(fitz.csRGB, pix)
```

**Warning signs:** `RuntimeError: cannot save to PNG` or washed-out/inverted colours in output.

### Pitfall 2: Ollama Vision Timeout on First Call (Cold Start)

**What goes wrong:** LLaVA requires loading the vision model into memory. First call on a cold Ollama instance can take 30–90 seconds for a 7B model. A tight timeout causes false failures.

**Why it happens:** LLaVA 7B is ~4GB; Ollama loads it on demand. Unlike the 0.5B judge model used in the hook, LLaVA load time is substantial.

**How to avoid:** Use a generous timeout (60s default, configurable). The fail-open pattern (D-11) means a timeout skips the image but doesn't break the embed run. Log clearly to stderr: `"Warning: vision model timeout for page N image M — skipping"`.

**Warning signs:** `image_count > 0` and `image_chunks == 0` in sidecar after embed (exactly the signal D-14 defines).

### Pitfall 3: `page.get_images()` vs `page.get_image_info()` Confusion

**What goes wrong:** `page.get_image_info()` exists on `fitz.Page` (verified in API inspection) but returns different data from `page.get_images()`. `get_images()` returns a list of tuples `(xref, smask, width, height, bpc, colorspace, alt_colorspace, name, filter, referencer)`.

**Why it happens:** PyMuPDF has multiple image-related methods added across versions with overlapping names.

**How to avoid:** Use `page.get_images(full=True)` — `img[0]` is the xref. Do not use `get_image_info()` for xref extraction.

### Pitfall 4: Vector-Only Pages Produce Huge Pixmaps at High DPI

**What goes wrong:** A full A4 page rendered at 300 DPI is 2480×3508 px (~25MB uncompressed). Sending this to a vision model is slow and may exceed the model's context.

**Why it happens:** Page-render fallback renders the entire page, not just the diagram area.

**How to avoid:** D-05 specifies 150 DPI for the fallback render. At 150 DPI an A4 page is 1240×1754 px — manageable. Do not increase this default.

### Pitfall 5: `chunk_index` Collisions Between Text and Image Chunks

**What goes wrong:** Image description chunks assigned `chunk_index` values starting at 0 collide with text chunks from the same document in `_point_id()` / `_point_id_versioned()`, causing Qdrant upsert to overwrite text chunks with image chunks.

**Why it happens:** `_point_id` is keyed on `slug + chunk_index`. If both text and image chunks use the same slug with overlapping chunk_index space, they produce the same UUID.

**How to avoid:** Start image chunk indices at `len(raw_chunks)` so the index space is contiguous and non-overlapping. Alternatively, include `doc_type` in the point ID hash — but the simpler offset approach works without changing `embed.py`.

### Pitfall 6: `get_drawings()` Returns Paths for Thin Borders and Rules

**What goes wrong:** Nearly every PDF page has vector drawings — page borders, table rules, separator lines. Using `len(page.get_drawings()) > 0` as the trigger for page-render fallback would render every page.

**Why it happens:** `get_drawings()` captures all vector paths, including trivial ones.

**How to avoid:** Only apply the page-render fallback when the page has zero embedded image objects (from `page.get_images()`) AND `page.get_drawings()` returns more than a threshold of paths (e.g., ≥ 5), or alternatively always apply the fallback only if there are no embedded images AND the page has non-trivial drawing bbox area. The simplest heuristic: trigger fallback only when `page.get_images()` is empty AND `len(page.get_drawings()) >= 3`. This correctly skips text-only pages with a simple border.

## Open Questions

1. **Does `fitz.Pixmap` have a `.scale()` method in 1.27.2?**
   - What we know: PyMuPDF 1.27.2 is installed; API inspection shows `Pixmap` methods but `.scale()` wasn't enumerated.
   - What's unclear: How to resize an extracted embedded image Pixmap without PIL.
   - Recommendation: At implementation time, run `python3 -c "import fitz; print([m for m in dir(fitz.Pixmap) if 'scale' in m.lower()])"`. If `.scale()` missing, use `fitz.Matrix(scale, scale)` with `page.get_pixmap(clip=img_bbox, matrix=mat)` as the scaling mechanism.

2. **Is `ollama_vision_model` needed in the `embed` sub-dict or at top level?**
   - What we know: `ollama_model` is in `DEFAULTS["embed"]["ollama_model"]`. D-03 says follow that pattern.
   - Recommendation: Add `ollama_vision_model` to `DEFAULTS["embed"]` sub-dict: `DEFAULTS["embed"]["ollama_vision_model"] = "llava:latest"`.

3. **Does the Ollama LLaVA model require any special pull command?**
   - What we know: `ollama pull llava` pulls the model. `llava:latest` maps to LLaVA 1.6 in current Ollama.
   - Recommendation: Document in `carta init` output or README that `ollama pull llava` is required for vision features. Do not block embed on model absence — fail-open (D-11).
