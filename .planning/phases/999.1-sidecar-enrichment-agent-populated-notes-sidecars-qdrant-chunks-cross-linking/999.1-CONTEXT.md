# Phase 999.1: Document Versioning, Stale Memory Tracking & Chunk Lifecycle — Context

**Gathered:** 2026-03-28
**Status:** Ready for planning

---

## Phase Goal (Revised)

This phase expands the original 999.1 scope (agent-populated notes + cross-linking) to include the foundational versioning and lifecycle infrastructure those features depend on. The revised goal is:

> Each embedded document has a content hash that tracks mutations. Chunks in Qdrant are linked to their source sidecar and carry generation/staleness metadata. When documents change, the system marks them stale. When documents are deleted or superseded, chunks are orphaned and eventually cleaned. Claude Code (via MCP) can autonomously trigger re-embedding.

---

## Locked Decisions

### 1. Hash Strategy — SHA256 + LF Normalization

- **Algorithm:** SHA256
- **Normalization:** CRLF → LF before hashing (binary read + `replace(b'\r\n', b'\n')`)
- **Rationale:** Cross-platform stability. Git normalizes line endings; we mirror that behavior for non-git-tracked files.
- **Applies to:** Markdown files. PDFs: hash raw bytes (no normalization needed, binary format).
- **New sidecar field:** `file_hash` (hex string), `hash_algorithm: sha256`

### 2. Change Detection — mtime Fast Path + Hash on Mismatch

- **On each `carta embed` run:**
  1. Check `file_mtime` first (cheap). If unchanged → skip hash computation.
  2. If mtime changed → compute hash, compare with stored `file_hash`.
  3. If hash changed → mark file as `stale`, bump `generation`, update `last_hash_check_at`.
  4. If hash unchanged despite mtime change → update stored mtime only (no stale flag).
- **New sidecar fields:** `last_hash_check_at` (ISO timestamp)

### 3. Status Taxonomy — Three Distinct States

Files and chunks have distinct classification from existing `status` field:

| Status | Meaning |
|--------|---------|
| `pending` | Not yet embedded (existing) |
| `embedded` | Currently embedded, hash matches latest file |
| `stale` | File content changed since last embed (hash mismatch) |
| `superseded` | Older generation of a doc that has been re-embedded (still valid, just older) |
| `orphaned` | Chunk has no matching sidecar/version; queued for cleanup |

- **Stale** = content changed, re-embed needed
- **Superseded** = older generation of a living document
- **Orphaned** = no anchor (sidecar deleted, or hash doesn't match any known version, or N generations behind threshold)
- These are stored in **both** sidecar (`status`) and Qdrant chunk payload.

### 4. Staleness Alert Threshold

- **Collection-wide staleness threshold:** 30% (configurable in `.carta/config.yaml` under `embed.stale_alert_threshold`)
- **Behavior when threshold crossed:** Print alert to stdout at end of `carta embed` run: `"⚠ 32% of embedded docs are stale — run 'carta embed --force-stale' to refresh"`
- **Hook behavior:** At retrieval time, if a returned chunk has `stale: true` in payload, Ollama judge appends a note: `[⚠ sourced from stale doc — run carta embed to refresh]`
- **Claude's Discretion:** Exact warning message wording.

### 5. Search-Time Stale Behavior — Warning Note, Not Filtering

- **Option A selected:** Return stale chunks with a warning note appended.
- **Rationale:** Stale docs may still contain valid information. Don't assume the user wants them filtered. Let them decide.
- **Implementation:** Hook injects warning note into context when `stale_as_of` is non-null in chunk payload.
- **Claude Code agent autonomy:** Claude Code (via `carta_embed` MCP tool) can autonomously choose to trigger re-embedding when stale docs are surfaced. This is a first-class use case — the MCP tool must support per-file or per-collection re-embed calls, not just full-collection runs.

### 6. Orphan Detection — All Three Triggers (Option C)

A chunk becomes `orphaned` when ANY of these are true:
1. Its source sidecar file is deleted
2. Its `doc_generation` is more than `embed.max_generations` (default: 2) behind current generation
3. Its `file_hash` doesn't match any known version hash in the sidecar history

**Cleanup timing:** Orphaned chunks are marked immediately but physically deleted from Qdrant after `embed.orphan_ttl_days` (default: 30 days).

**Exception — protected collections:** Chunks in `quirk`, `bug-note`, or `helpful-note` doc_type collections are **exempt from orphan TTL** and must be explicitly deleted. They survive indefinitely unless manually removed.

### 7. Chunk Retention — Rolling Generation Window

- **Policy:** Keep last N generations per document (default: 2, configurable as `embed.max_generations`)
- **On re-embed:** Old chunks are marked `superseded` (not deleted immediately). After cleanup pass, superseded chunks beyond generation window are marked `orphaned` and enter 30-day TTL.
- **Exception:** Protected doc_types (`quirk`, `bug-note`, `helpful-note`) never enter the rolling window — they are version-independent.

### 8. Qdrant Chunk Payload Schema — New Fields

All chunks upserted after this phase must include these additional payload fields:

```yaml
doc_generation: 1          # integer, increments on each re-embed
stale_as_of: null          # ISO timestamp or null — when hash mismatch detected
superseded_at: null        # ISO timestamp or null — when newer generation created
orphaned_at: null          # ISO timestamp or null — when orphaned
sidecar_id: "path/to/doc.embed-meta.yaml"  # relative path back to source sidecar
chunk_source_hash: "abc123..."             # file_hash at time of embedding
```

- These fields are **scannable by Python tooling and Ollama judge** but should not be injected into agent context wholesale.
- The judge reads them to annotate results; Python scan tools use them for cleanup.

### 9. Doc Type / Memory Type Taxonomy

Introduce a broader `doc_type` taxonomy for all Qdrant collections:

| Type | Description | Orphan Exempt? |
|------|-------------|----------------|
| `doc` | Standard embedded document (PDF, markdown) | No |
| `datasheet` | Product/component spec sheet | No |
| `manual` | User/technical manual | No |
| `quirk` | Non-obvious behavior note, "save our life in 2 years" notes | **Yes** |
| `bug-note` | Known bug documentation, workarounds | **Yes** |
| `helpful-note` | General helpful memory not tied to a specific doc | **Yes** |
| `session` | Session memory (existing) | N/A |

- Protected types (`quirk`, `bug-note`, `helpful-note`) use a separate Qdrant collection: `{project}_notes`
- These are embedded via the same pipeline but skip the stale/orphan lifecycle.

### 10. Sidecar Schema — Updated Fields

Full updated sidecar stub including new fields:

```yaml
slug: "chip-datasheet"
doc_type: "datasheet"
file_type: "pdf"             # or "markdown"
current_path: "docs/chips/chip.pdf"
status: "embedded"           # pending | embedded | stale | superseded | orphaned
indexed_at: "2026-03-28T..."
chunk_count: 14
file_mtime: 1743120000.0
file_hash: "abc123..."       # SHA256, LF-normalized for text; raw for PDF
hash_algorithm: "sha256"
generation: 1                # increments on each successful re-embed
last_hash_check_at: "2026-03-28T..."
collection: "project_doc"
spec_summary: null
notes: ""
version_history:             # optional: last N file_hash values
  - hash: "abc123..."
    generation: 1
    indexed_at: "2026-03-28T..."
```

### 11. Claude Code Agent Autonomy via MCP

- `carta_embed` MCP tool must support granular invocation:
  - `carta_embed(scope="stale")` — re-embed only stale files
  - `carta_embed(scope="file", path="docs/chip.pdf")` — re-embed single file
  - `carta_embed(scope="all")` — full re-embed (existing behavior)
- This enables Claude Code to self-manage doc freshness mid-session without human intervention.
- Claude Code can combine `carta_scan` (discover stale) + `carta_embed(scope="stale")` in an autonomous loop.
