# Phase 999.4 — GLM-OCR Intelligent Extraction — Context

## Overview

**Goal:** Enhance Carta's PDF extraction with intelligent content classification that routes pages to the optimal vision model (GLM-OCR for text/tables, LLaVA for visuals).

**Why this matters:**
- Current LLaVA extraction paraphrases text and misses table structure
- GLM-OCR excels at exact text extraction and markdown table preservation
- Technical docs (datasheets, manuals) benefit massively from structured OCR
- Visual content (schematics, plots) still needs LLaVA's descriptive power
- Automatic routing means users don't need to think about it

## Current State (Post-999.2)

- Vision pipeline exists: PyMuPDF extracts page images
- Single model: `ollama_vision_model` (typically LLaVA)
- Sidecars track `vision_status` and `image_chunks`
- Images embedded as chunks with LLaVA descriptions

## Target State (Post-999.4)

- Dual model support: LLaVA + GLM-OCR
- Automatic content classification per-page
- Text-heavy pages → GLM-OCR (exact extraction)
- Visual pages → LLaVA (descriptions)
- Mixed pages → Hybrid (both, combined)
- Enhanced sidecars track which model extracted what
- Structured chunking preserves table integrity

## Key Design Decisions

### 1. Classification Strategy
- **Heuristic-based** (not ML) for speed and determinism
- Uses PyMuPDF text block analysis + image density
- Thresholds: >70% text = TEXT, <30% text + >40% image = VISUAL
- Table detection via aligned x-coordinate patterns

### 2. Model Routing
- Single-pass classification before extraction
- No user configuration required (automatic)
- Optional config for power users: `vision_routing: auto|ocr|vision|both`

### 3. Chunking Changes
- GLM-OCR output is markdown (tables as `| col |`)
- Chunker detects markdown tables, keeps them whole
- Text sections still respect `max_tokens`
- Visual descriptions chunked normally

### 4. Sidecar Evolution
- `vision.pages_analyzed` → `vision.extraction_summary`
- Per-page breakdown: model used, content type, table count
- Chunk-level metadata: `source_model`, `chunk_type`

## Technical Stack

| Component | Technology |
|-----------|------------|
| PDF parsing | PyMuPDF (fitz) |
| Text OCR | GLM-OCR via Ollama (`glm-ocr:latest`) |
| Visual description | LLaVA via Ollama (`llava:latest` or user choice) |
| Classification | Heuristics (no ML model) |
| Chunking | Enhanced text splitter with table preservation |

### What users see
- Better search results for specific values ("pin 47 is VCC")
- Tables searchable by any cell value
- No change to visual diagram search
- Sidecars show extraction provenance

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Classification wrong | Hybrid mode for mixed pages; user can force mode |
| GLM-OCR unavailable | Falls back to LLaVA with warning |
| Slower extraction | GLM-OCR is faster than LLaVA; net neutral |
| Table preservation fails | Configurable `preserve_tables: false` to disable |
