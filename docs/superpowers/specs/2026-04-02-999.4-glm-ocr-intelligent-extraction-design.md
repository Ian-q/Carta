# Phase 999.4 — GLM-OCR Intelligent PDF Extraction — Design Spec

**Project:** Carta (`carta-cc`)  
**Date:** 2026-04-02  
**Status:** Draft (ready for planning)  
**Depends on:** Phase 999.2 (Vision pipeline established)

## Goal

Enhance Carta's PDF extraction with **intelligent content classification** that routes pages to the optimal vision model:

- **Text-heavy pages** (datasheets, tables, specs) → **GLM-OCR** for exact structured extraction
- **Visual pages** (plots, schematics, diagrams) → **LLaVA** for descriptive context
- **Mixed pages** → Hybrid approach with both models

This maximizes extraction quality while maintaining the existing embedding pipeline.

## Non-Goals

- Replacing LLaVA entirely (both models have strengths)
- Real-time OCR streaming (batch PDF processing only)
- Handwriting recognition (focus on printed technical documents)
- Multi-language OCR beyond English (GLM-OCR supports Chinese, but scope to EN first)

## Current Baseline (Context)

- Phase 999.2 established vision pipeline with PyMuPDF + Ollama vision models
- `ollama_vision_model` config option exists (defaults to LLaVA)
- Images are extracted per-page, described with vision model, embedded as chunks
- Sidecars track `vision_status` and `image_chunks`

## Content Classification Model

### Classification dimensions

| Dimension | Values | Detection method |
|-----------|--------|------------------|
| **text_density** | high / medium / low | PyMuPDF text extraction coverage |
| **table_presence** | yes / no | Structural pattern matching |
| **visual_elements** | diagrams / plots / photos / none | Image density + contour analysis |
| **content_type** | text / mixed / visual | Composite score from above |

### Classification heuristics

```python
def classify_page_content(page) -> ContentType:
    """
    Analyze PDF page to determine optimal extraction strategy.
    
    Returns ContentType:
        TEXT:    >70% text coverage OR table structures detected
        MIXED:   30-70% text coverage with visual elements
        VISUAL:  <30% text coverage, image-heavy
    """
    # 1. Text density via PyMuPDF
    text_blocks = page.get_text("blocks")
    text_area = sum(block[2] * block[3] for block in text_blocks)  # width * height
    page_area = page.rect.width * page.rect.height
    text_coverage = text_area / page_area
    
    # 2. Table detection (markdown-like structures)
    has_tables = detect_table_patterns(text_blocks)
    
    # 3. Image density
    image_list = page.get_images()
    image_coverage = sum(img[2] * img[3] for img in image_list) / page_area
    
    # 4. Composite classification
    if text_coverage > 0.70 or has_tables:
        return ContentType.TEXT
    elif text_coverage < 0.30 and image_coverage > 0.40:
        return ContentType.VISUAL
    else:
        return ContentType.MIXED
```

## Dual Extraction Pipeline

### Model routing logic

```python
class ExtractionRouter:
    """Routes PDF pages to appropriate vision model based on content classification."""
    
    def __init__(self, cfg: dict):
        self.ocr_model = cfg["embed"].get("ocr_model", "glm-ocr:latest")
        self.vision_model = cfg["embed"].get("ollama_vision_model", "llava:latest")
        self.classifier = ContentClassifier()
    
    def extract_page(self, page_image: bytes, page_num: int) -> ExtractionResult:
        # Classify content type
        content_type = self.classifier.classify(page_image)
        
        # Route to appropriate model
        if content_type == ContentType.TEXT:
            return self._extract_with_ocr(page_image, page_num)
        elif content_type == ContentType.MIXED:
            return self._extract_mixed(page_image, page_num)
        else:  # VISUAL
            return self._extract_with_vision(page_image, page_num)
    
    def _extract_with_ocr(self, image: bytes, page: int) -> ExtractionResult:
        """Use GLM-OCR for structured text/tables."""
        response = ollama.generate(
            model=self.ocr_model,
            prompt="""Extract all text content from this document page.
            
If tables are present, format them as markdown tables with proper headers.
Preserve numerical values, units, and specifications exactly as shown.
Maintain the document's hierarchical structure (headers, lists, paragraphs).

Output format:
- Main content as plain text or markdown
- Tables as markdown tables
- No explanatory text, just the extracted content""",
            images=[image]
        )
        
        return ExtractionResult(
            text=response['response'],
            source="glm-ocr",
            content_type="text",
            structured=True
        )
    
    def _extract_with_vision(self, image: bytes, page: int) -> ExtractionResult:
        """Use LLaVA for visual content description."""
        response = ollama.generate(
            model=self.vision_model,
            prompt="Describe this technical diagram for engineering documentation search.",
            images=[image]
        )
        
        return ExtractionResult(
            text=response['response'],
            source="llava",
            content_type="visual",
            structured=False
        )
    
    def _extract_mixed(self, image: bytes, page: int) -> ExtractionResult:
        """For mixed content, use both models and combine."""
        ocr_result = self._extract_with_ocr(image, page)
        vision_result = self._extract_with_vision(image, page)
        
        # Combine: OCR for text, vision for context
        combined = f"""{ocr_result.text}

[Visual Context]: {vision_result.text}"""
        
        return ExtractionResult(
            text=combined,
            source="hybrid",
            content_type="mixed",
            structured=ocr_result.structured
        )
```

## Structured Output Handling

### GLM-OCR output format

GLM-OCR returns markdown-compatible text with:
- Headers preserved (`#`, `##`)
- Tables as markdown tables (`| col1 | col2 |`)
- Lists as markdown lists (`-`, `1.`)
- Code/specs as code blocks or plain text

### Chunking strategy for structured content

```python
def chunk_structured_ocr(text: str, max_tokens: int = 800) -> list[Chunk]:
    """
    Chunk OCR output preserving table integrity.
    
    Strategy:
    1. Split by markdown headers (##) first
    2. Keep tables intact (don't split mid-table)
    3. Respect max_tokens boundary for text sections
    """
    chunks = []
    
    # Pattern: Split on headers but keep tables whole
    sections = re.split(r'\n(?=##?\s)', text)
    
    for section in sections:
        if is_markdown_table(section):
            # Tables stay as single chunks (even if large)
            chunks.append(Chunk(
                text=section,
                chunk_type="table",
                estimated_tokens=estimate_tokens(section)
            ))
        else:
            # Text sections respect token limits
            for text_chunk in chunk_text(section, max_tokens):
                chunks.append(Chunk(
                    text=text_chunk,
                    chunk_type="text",
                    estimated_tokens=estimate_tokens(text_chunk)
                ))
    
    return chunks
```

### Migration path

Existing configs without `ocr_model` will:
1. Continue using `ollama_vision_model` for all pages (backward compatible)
2. Log a warning: "GLM-OCR not configured, using LLaVA for all extraction"
3. User can opt-in by adding `ocr_model: glm-ocr:latest`

## Performance Considerations

### Speed

| Scenario | LLaVA-only | Dual (GLM-OCR + LLaVA) | Impact |
|----------|------------|------------------------|--------|
| 10-page text PDF | 10 calls | 10 calls (GLM-OCR) | ~20% faster (GLM-OCR is faster) |
| 10-page mixed PDF | 10 calls | 10 calls (split) | Similar |
| 10-page visual PDF | 10 calls | 10 calls (LLaVA) | Same |

**Net result**: No significant slowdown; GLM-OCR is actually faster for text pages.

### Memory
- Classification happens per-page, streaming
- No full PDF loaded in memory simultaneously
- Peak memory: single page image + model context

### Ollama resource management
- Both models use ~2-4GB VRAM each
- Sequential calls (not parallel) to avoid resource contention
- Recommend: Ollama with 8GB+ VRAM for comfortable operation

## Success Metrics

1. **Extraction quality**: GLM-OCR extracts register tables with >95% accuracy vs LLaVA's ~70%
2. **RAG improvement**: Search queries for specific values ("temperature range", "pin 47 function") return correct documents more often
3. **User transparency**: Sidecars clearly show which model extracted which content
4. **Zero regression**: Visual diagrams still searchable via LLaVA descriptions

## Dependencies

- PyMuPDF (existing) - for PDF parsing and image extraction
- Ollama with `glm-ocr:latest` - for text/table OCR
- Ollama with `llava:latest` (or other) - for visual description
- Python 3.10+ (existing)

## Future Extensions (Out of Scope for 999.4)

- **Multi-language**: Leverage GLM-OCR's Chinese support for bilingual docs
- **Streaming**: Real-time OCR as pages are scrolled (for large PDFs)
- **Custom prompts**: Per-document-type prompts (e.g., "extract BOM tables specifically")
- **Confidence scoring**: Model confidence to flag uncertain extractions
