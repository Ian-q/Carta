# Phase 999.3 — Qdrant Collection Scoping (Repo Isolation + Shared + Global) — Design Spec

**Project:** Carta (`carta-cc`)  
**Date:** 2026-03-31  
**Status:** Draft (approved for planning)  

## Goal

Make Carta’s retrieval **repo-isolated by default** (no cross-repo contamination), while enabling **explicit opt-in** to:

- **shared**: recall across an allowlisted set of projects/repos
- **global**: recall from an explicit, promoted global pool (`carta_global_*`)

This applies consistently across **CLI search**, **MCP search**, and the **prompt hook**.

## Non-Goals

- Multi-tenant auth / access control (local tool)
- Automatic global promotion (promotion UX can come later)
- Changing embedding semantics or chunk schema beyond what scoping requires

## Current Baseline (Context)

- Collections are project-namespaced: `{project_name}_{type}` (e.g. `_doc`, `_session`, `_notes`)
- Config already has `cross_project_recall` scaffolding in `.carta/config.yaml` and defaults
- `999.1` introduced doc-type routing primitives (e.g. protected doc types routed to notes collections)

## Scope Model

### Scope levels

- **repo** (default): query only the current repo’s collections, derived from the current repo’s config
- **shared** (opt-in): query multiple repos’ collections based on an **explicit allowlist** of projects
- **global** (opt-in): additionally query global collections: `carta_global_{doc|notes|session}` (exact set defined below)

### Safety invariants

- **Default = repo-only** even if other collections exist in Qdrant.
- **Shared requires allowlist**: never “search everything” by default.
- **Hook remains conservative**: hook uses config default scope; should never silently widen beyond user-configured opt-in.
- **Shared/global always include current project**: when scope is widened, results still include the current repo’s project unless an explicit “only these projects” option is introduced (out of scope for 999.3).

## Collection Naming & Routing

### Existing

- Repo/project collections:
  - `{project}_doc`
  - `{project}_session`
  - `{project}_notes` (protected doc types: `quirk`, `bug-note`, `helpful-note`)

### New: global collections

Introduce global collections (project-agnostic):

- `carta_global_doc`
- `carta_global_notes`
- `carta_global_session` (optional; default off unless a later phase needs it)

**Recommendation:** implement `doc` + `notes` first. Session global recall is likely noisy.

## Configuration Spec

### Canonical config (single switch)

Use a single canonical switch for cross-project recall:

```yaml
cross_project_recall:
  mode: repo        # repo | shared | global
  projects_allowlist: []  # only used for shared/global; always includes current project implicitly
```

Notes:
- `mode=shared` requires `projects_allowlist` non-empty.
- `mode=global` additionally queries global collections (see below).

### Identity and collision note

Allowlist entries are **project identifiers**. Initially, this spec uses `project_name` (existing config key) as the identifier.

- **Risk:** `project_name` collisions across unrelated repos can reintroduce cross-repo contamination.
- **Mitigation (required in planning):** add a stable `project_id` option (e.g. hash of git remote URL, or explicit user-defined id) and store it in payload for attribution and filtering.

## API / Interface Changes

### CLI

Add an optional scope override to `carta search`:

- `carta search "query"` → uses config-derived scope (default `repo`)
- `carta search --scope repo|shared|global "query"` → overrides config for that invocation
- `carta search --scope shared --projects p1,p2 "query"` → per-call allowlist override for shared/global

**Override semantics:**
- If `--projects` is provided, it **overrides** config allowlist for that call (and still implicitly includes the current project).
- If `--projects` is omitted, the config allowlist is used.

### MCP

Extend `carta_search` tool schema to accept:

- `scope: "repo" | "shared" | "global"` (optional)
- `projects: list[str]` (optional; only used for shared)

The MCP server must:
- Validate scope parameters and return structured error dicts (no exceptions)
- Never print to stdout (JSON-RPC cleanliness)

**Error contract (minimum):**

```json
{
  "ok": false,
  "error": {
    "code": "invalid_request",
    "message": "shared scope requires projects_allowlist",
    "details": {"scope": "shared"}
  }
}
```

### Hook

Hook uses config scope only:
- default `repo`
- if `cross_project_recall.mode` is `shared` or `global`, it follows that (still allowlist-gated)

**Config resolution (must be explicit):**
- Hook resolves config by walking up from the repo working directory to find `.carta/config.yaml` (existing behavior via `find_config`).
- If no config is found, hook must behave as `repo` scope with cross-project recall disabled.

## Search Pipeline Behavior

Introduce a single “scope resolution” decision point so all callers behave the same:

1. Determine repo `project_name` (current config)
2. Determine effective scope:
   - if `scope override` provided (CLI/MCP) → validate and use
   - else → use config scope (`repo` unless cross-project recall enabled)
3. Produce collection list to query:
   - repo: `[f"{project}_doc", f"{project}_notes", f"{project}_session" ...]` (as appropriate for doc_types searched)
   - shared: union over allowlisted projects **plus current project**
   - global: shared (or repo) plus `carta_global_doc` + `carta_global_notes` (session global excluded by default)
4. Query Qdrant across these collections and return results with clear attribution:
   - include `collection` for every hit
   - include a stable project identifier in payload for disambiguation (planning requirement; see identity note above)

### Doc-type → collection routing table (required)

Define a deterministic mapping so repo/shared/global behave identically:

- **doc-like types** → query `{project}_doc` (and `carta_global_doc` in global)
- **protected note types** (`quirk`, `bug-note`, `helpful-note`) → query `{project}_notes` (and `carta_global_notes` in global)
- **session** → query `{project}_session` (global session excluded by default)

## Multi-Repo Test Matrix (Acceptance)

Use two repos with distinct project identifiers and distinct documents embedded.

### Tests

1. **Repo isolation default**
   - In Repo A: `carta search "unique-from-b"` returns no results from Repo B
2. **Shared opt-in**
   - In Repo A with allowlist `[B]`: search can return B hits, clearly attributed
3. **Global pool**
   - With global enabled + promoted global content embedded into `carta_global_*`, `--scope global` returns it; `--scope repo` does not
4. **Hook safety**
   - Hook does not widen scope unless config explicitly enables it

### Edge cases (required)

- **Shared with empty allowlist:** returns `invalid_request` (CLI error / MCP error object), does not fall back silently.
- **Override precedence:** CLI/MCP overrides must not mutate config; they apply to that call only.
- **Collision case:** two repos with same `project_name` must be treated as a hazard; planning must implement/require stable `project_id` to prevent contamination.

## Rollout / Migration Notes

- Backwards compatible: default behavior becomes stricter, but only affects users who previously relied on accidental cross-repo collisions.
- No changes required to existing per-project collections.
- Global collections are additive and can be created lazily on first use.

## Open Questions (deferred to planning)

- Whether global session memories are desirable (`carta_global_session`)
- How promotion into global should work (CLI command vs MCP tool vs manual)
- Stable identity: how to define and persist `project_id` for allowlisting and attribution (recommended) vs using `project_name` only (hazard).
- For 999.3 testing, global collections can be seeded manually (direct Qdrant insert or a dev-only helper) if no promotion UX exists yet.
